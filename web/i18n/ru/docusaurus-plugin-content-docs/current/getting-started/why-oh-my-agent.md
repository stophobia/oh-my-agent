# Почему oh-my-agent

Категория multi-agent CLI уже насыщена. Только за последний квартал появилось более двадцати multi-agent orchestrator-ов: Metateam, OpenSwarm, DevSquad, Praktor, Salacia, Codelegate, agent-of-empires, TTal, Maggy и другие. Большинство оптимизируют одну и ту же ось - заставить агентов писать код быстрее.

oh-my-agent оптимизирует другую ось. Исходная гипотеза: при достаточно мощных моделях затраты на анализ, проектирование и реализацию в SDLC стремятся к нулю. Дорогая часть разработки ПО всегда была тестированием и поддержкой - сохранением системы рабочей, безопасной и понятной после первого commit-а. Именно на этой оси построен oh-my-agent.

Эта страница конкретизирует это позиционирование. Развёрнутую дискуссию, породившую этот фрейм, см. в [issue #155](https://github.com/first-fluke/oh-my-agent/issues/155#issuecomment-4142133589).

## Ось затрат сместилась

Когда одна мощная модель производит работающую фичу за минуты, узкое место - больше не throughput реализации. Узкое место - верифицировать, что произведённый код действительно делает то, что он заявляет; ловить тихие регрессии между итерациями; держать секреты вне prompt-ов и логов; делать видимыми расходы на токены до того, как они удивят команду.

Harness, который просто spawn-ит агентов быстрее, не решает ничего из этого. Harness, спроектированный для пост-реализационной фазы - решает.

## Что oh-my-agent даёт настоящему центру затрат

Каждая возможность ниже отвечает на конкретный режим отказа, отмеченный в категории multi-agent CLI.

### Независимая верификация, а не самооценка LLM

`oma verify <agent>` выполняет четырнадцать детерминированных проверок на тип агента. Все проверки механические: exit code тестовой команды, TypeScript strict пройден, обнаружение паттернов raw SQL, сканирование захардкоженных секретов, Flutter analyze, сканирование inline styles, нарушение scope относительно charter-а агента. Никакой LLM не судит, "выглядит ли работа корректной". Проверка проходит тогда и только тогда, когда базовая команда сообщает об успехе.

Это отвечает на самую частую жалобу в категории, обобщённую в одном сообществе как "agents lie - they say tests pass when tests do not". Список проверок см. в `cli/commands/verify/verify.ts`.

### Повторная верификация между итерациями

Workflow `ralph` оборачивает `ultrawork` независимой фазой JUDGE. После каждой итерации JUDGE повторно верифицирует каждый criterion - включая те, что уже прошли в предыдущих итерациях. Это ловит случай, когда исправление C2 тихо ломает C1, что и есть реальный механизм большинства регрессий в длинных сессиях агентов.

Тяжёлые верификации (более тридцати секунд) кешируются против затронутых путей файлов, чтобы повторная верификация оставалась дешёвой. Полный протокол см. в `.agents/workflows/ralph/resources/judge-protocol.md`.

### Quota cap, блокирующие до ущерба

Каждый вызов `oma agent:spawn` записывает оценку токенов этого spawn-а в `.serena/memories/session-cost-{sessionId}.md`. Перед следующим spawn-ом `checkCap` обращается к настроенному quota cap-у и отказывается запускать, если какое-либо измерение превышено. Принудительно применяются три измерения: всего токенов, всего spawn-ов, бюджет токенов по vendor-у.

Это разница между узнаванием постфактум, что потрачено сорок тысяч долларов, и получением на пятнадцатом spawn-е предупреждения, что в бюджете остался один spawn. См. `cli/io/session-cost.ts` и настройте под `session.quota_cap` в `.agents/oma-config.yaml`.

### Retry, затем explore, а не retry навсегда

Когда `orchestrate` Step 5 находит ошибку верификации, он повторяет агента максимум два раза с контекстом ошибки. Если второй retry всё ещё проваливается, а cost cap ещё не превышен, workflow переключается в Exploration Loop: параллельно spawn-ит две или три альтернативные варианты гипотез в отдельных workspace-ах и оставляет только результат с наивысшим скором. Неудачные подходы отбрасываются с зарегистрированной стоимостью.

Это структурированный ответ на случай, когда подход фундаментально неверен. Повторение того же никогда не сходится; пробование разных подходов параллельно - сходится.

### Monorepo-осведомлённый routing workspace

`detectWorkspace` читает конфигурации pnpm, nx, turbo и lerna и направляет каждого агента в соответствующий sub-workspace автоматически. Backend agent работает против `apps/api/`, frontend agent против `apps/web/`, без необходимости orchestrator-у собирать пути вручную. См. `cli/io/workspaces.ts`.

## Multi-vendor не опционален

Вторая проектная гипотеза: любая команда, серьёзно занимающаяся AI-ассистированной разработкой, использует более одного провайдера. Сегодня это означает Claude, Codex, Gemini, Copilot, Qwen, Kimi и то, что появится в следующем квартале. Смена vendor-а - факт, а не edge case: Anthropic переносит агентные функции в отдельный платный plan, OpenAI выпускает Codex CLI в ту же неделю, когда модели Anthropic деградируют, GitHub Copilot переходит на consumption-based pricing.

oh-my-agent трактует выбор vendor-а как конфигурацию per-agent через `model_preset` и `agents.<id>.model` в `.agents/oma-config.yaml`. Переносимая директория `.agents/` - это single source of truth; каждый поддерживаемый runtime проецирует из неё. Lock-in vendor-а не нужен для использования oh-my-agent, и миграция не нужна при смене.

## Repo-native кастомизация

Третья гипотеза: ни две команды не разделяют одно определение "done". Одна команда требует сканирования OWASP Top 10 при каждом изменении backend-а. Другая требует QA-отчёт на корейском. Третья требует, чтобы каждая migration ревьюилась database agent-ом перед merge-ем.

Поскольку `.agents/` - это просто файлы в вашем репозитории, каждая команда может добавлять или модифицировать агентов, skill-ы, workflow-ы и quality gate-ы, чтобы соответствовать собственному кодексу поведения и compliance-позиции. Кастомизация - это `git commit`, а не тикет поддержки vendor-а.

## Что это значит на практике

Если ваш приоритет - "spawn-ить параллельных агентов быстро", многие инструменты покрывают эту поверхность. Если ваш приоритет - "доставить код, который продолжает работать после того, как агенты ушли из комнаты", oh-my-agent построен именно для этой цели. `oma verify`, JUDGE, Exploration Loop, quota cap и monorepo routing - не опциональные add-on-ы, а причина существования проекта.

Для подробностей по каждой возможности см. раздел Core Concepts (Agents, Parallel Execution) в боковой панели.
